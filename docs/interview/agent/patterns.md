## 题目1: Agent的核心组件 ⭐

**题型**: 简答题  
**难度**: ⭐⭐  
**知识点**: Agent基础

### 问题
一个完整的Agent系统包含哪些核心组件?

### 参考答案
**Agent的4大核心组件**:

1. **LLM(大脑)**
   - 负责推理、决策、生成
   - 选择合适的模型(GPT-4/Claude等)

2. **Planning(规划)**
   - 任务分解
   - 策略选择(ReAct/Plan-and-Execute)
   - 反思与修正

3. **Memory(记忆)**
   - 短期记忆: 当前对话上下文
   - 长期记忆: 向量数据库存储历史

4. **Tools(工具)**
   - 外部API调用(搜索、计算器)
   - 代码执行
   - 数据库查询

**工作流程**:
```
感知(Perception) → 思考(Thinking) → 行动(Action) → 观察(Observation) → 循环
```

### 评分标准
- 列出4个组件得4分
- 每个组件解释清晰得1分(共4分)
- 提到工作流程得2分
- **总分**: 10分,≥8分为合格

## 题目2: ReAct模式的原理 ⭐⭐

**题型**: 简答题  
**难度**: ⭐⭐⭐  
**知识点**: ReAct框架

### 问题
ReAct(Reasoning + Acting)模式的工作流程是什么?为什么它比纯推理更有效?

### 参考答案
**ReAct工作流程**:
```
Thought: 我需要查询天气信息
Action: search_api("北京今天天气")
Observation: 北京晴,20-25度
Thought: 我已获取天气信息,可以回答用户
Final Answer: 北京今天晴天,气温20-25度
```

**为什么更有效**:
1. **结合外部知识**: 通过工具获取实时信息
2. **可验证**: 每步观察可检查正确性
3. **减少幻觉**: 基于事实而非编造
4. **可解释**: 推理过程透明

**对比纯推理**:
- 纯推理: 依赖训练数据,可能过时或错误
- ReAct: 实时获取信息,更准确可靠

### 评分标准
- 描述工作流程得4分
- 解释优势得3分
- 对比分析得2分
- **总分**: 9分,≥7分为合格

## 题目3: Function Calling vs Tool Use ⭐

**题型**: 单选题  
**难度**: ⭐⭐  
**知识点**: Function Calling

### 问题
Function Calling和Tool Use的关系是什么?

A. 两者完全不同  
B. Function Calling是Tool Use的一种实现方式  
C. Tool Use是Function Calling的子集  
D. 只有OpenAI支持Function Calling  

**正确答案**: B

### 解析
**Function Calling**:
- OpenAI提出的结构化函数调用机制
- LLM输出JSON格式的工具调用请求
- 框架自动执行并返回结果

**Tool Use**:
- 更广义的概念,Agent使用外部工具
- 可通过Function Calling实现
- 也可通过Prompt工程手动解析

**关系**: Function Calling ⊂ Tool Use

### 评分标准
- 答对得5分

## 题目4: Agent的记忆管理策略 ⭐⭐

**题型**: 简答题  
**难度**: ⭐⭐⭐  
**知识点**: Agent记忆

### 问题
Agent如何管理短期记忆和长期记忆?各有什么技术方案?

### 参考答案
**短期记忆(Short-term Memory)**:
- **存储**: 内存中的对话历史
- **容量**: 受限于context window
- **优化**: 
  - 滑动窗口(保留最近N轮)
  - 摘要压缩(定期总结)
  - Token计数,超限截断

**长期记忆(Long-term Memory)**:
- **存储**: 向量数据库(PgVector/Milvus)
- **检索**: 相似度搜索相关历史
- **应用**:
  - 用户偏好记忆
  - 历史对话检索
  - 个性化推荐

**技术方案对比**:
| 类型 | 存储 | 检索速度 | 容量 |
|------|------|---------|------|
| 短期 | 内存/Redis | 快(ns级) | 小(KB-MB) |
| 长期 | Vector DB | 中(ms级) | 大(GB-TB) |

### 评分标准
- 短期记忆解释得3分
- 长期记忆解释得3分
- 技术方案对比得3分
- **总分**: 9分,≥7分为合格

## 题目5: Plan-and-Execute模式 ⭐⭐⭐

**题型**: 简答题  
**难度**: ⭐⭐⭐⭐  
**知识点**: Agent规划

### 问题
Plan-and-Execute模式与ReAct有什么区别?适用场景是什么?

### 参考答案
**Plan-and-Execute流程**:
```
1. Planner: 生成完整计划(步骤1,2,3...)
2. Executor: 按顺序执行每个步骤
3. Reviewer: 检查结果,必要时调整计划
```

**与ReAct的区别**:

| 维度 | ReAct | Plan-and-Execute |
|------|-------|------------------|
| 规划方式 | 逐步思考 | 预先完整规划 |
| 灵活性 | 高,动态调整 | 低,按计划执行 |
| 适用任务 | 探索性任务 | 结构化任务 |
| 错误恢复 | 即时修正 | 需重新规划 |

**Plan-and-Execute优势**:
- 适合复杂多步骤任务
- 可并行执行独立步骤
- 易于监控进度

**适用场景**:
- 报告生成(研究→写作→审查)
- 数据分析(清洗→分析→可视化)
- 工作流编排

### 评分标准
- 描述流程得3分
- 对比分析得4分
- 提到适用场景得2分
- **总分**: 9分,≥7分为合格

## 题目6: Multi-Agent协作的挑战 ⭐⭐⭐

**题型**: 简答题  
**难度**: ⭐⭐⭐⭐  
**知识点**: 多Agent系统

### 问题
多Agent协作系统面临哪些挑战?如何解决?

### 参考答案
**主要挑战**:

**1. 通信协议**
- 问题: Agent间如何传递消息
- 解决: 定义标准化消息格式(JSON),使用消息队列(RabbitMQ)

**2. 任务分配**
- 问题: 如何决定哪个Agent做什么
- 解决: Orchestrator统一调度,或基于能力路由

**3. 冲突解决**
- 问题: 多个Agent意见不一致
- 解决: 投票机制、优先级规则、人工仲裁

**4. 死锁检测**
- 问题: Agent互相等待导致停滞
- 解决: 超时机制、依赖图检测循环

**5. 状态一致性**
- 问题: 分布式环境下状态同步
- 解决: 共享数据库、事件溯源

**架构模式**:
- **集中式**: Orchestrator协调(易实现,单点故障)
- **去中心化**: Agent自主协商(灵活,复杂)

### 评分标准
- 列出4个以上挑战得4分
- 每个挑战给出解决方案得2分(共8分)
- 提到架构模式得2分
- **总分**: 14分,≥11分为合格

## 题目7: Agent的错误处理策略 ⭐⭐

**题型**: 多选题  
**难度**: ⭐⭐⭐  
**知识点**: Agent容错

### 问题
Agent执行过程中可能遇到哪些错误?如何应对?(多选)

A. API调用失败 → 重试机制  
B. Tool返回空结果 → 降级策略  
C. LLM输出格式错误 → 解析重试  
D. 无限循环 → 最大迭代次数限制  

**正确答案**: A、B、C、D

### 解析
**A. API调用失败** ✓
- 指数退避重试(1s, 2s, 4s, 8s)
- 最多重试3次,失败则报错

**B. Tool返回空结果** ✓
- 尝试其他Tool
- 告知用户信息不足
- 请求用户补充信息

**C. LLM输出格式错误** ✓
- 正则提取关键信息
- 重试并要求严格格式
- Few-shot示例强化格式

**D. 无限循环** ✓
- 设置max_iterations=5
- 检测重复动作
- 超时中断

### 评分标准
- 全选对得5分

## 题目8: Human-in-the-Loop的设计 ⭐⭐

**题型**: 简答题  
**难度**: ⭐⭐⭐  
**知识点**: 人工介入

### 问题
什么是Human-in-the-Loop?在什么场景下需要引入人工介入?

### 参考答案
**Human-in-the-Loop定义**:
- 在Agent工作流中设置人工审核节点
- 关键决策由人类确认

**需要人工介入的场景**:

**1. 低置信度决策**
- Agent置信度 < 阈值(如0.7)
- 示例: 意图识别不确定

**2. 高风险操作**
- 删除数据、转账、发布内容
- 需要人类最终确认

**3. 异常情况**
- Agent无法处理的边缘案例
- 多次重试仍失败

**4. 合规要求**
- 医疗诊断、法律建议
- 法规要求人工审核

**实现方式**:
- WebSocket推送待审核任务
- 人工界面审批/拒绝
- 回调通知Agent继续执行

### 评分标准
- 定义准确得3分
- 列出3个以上场景得4分
- 提到实现方式得2分
- **总分**: 9分,≥7分为合格

## 题目9: Agent评估指标 ⭐⭐⭐

**题型**: 简答题  
**难度**: ⭐⭐⭐⭐  
**知识点**: Agent评估

### 问题
如何评估一个Agent的性能?列举关键指标。

### 参考答案
**功能指标**:

**1. 任务完成率(Task Success Rate)**
- 成功完成任务的比例
- 目标: >85%

**2. 工具调用准确率**
- 选择正确Tool的比例
- 目标: >90%

**3. 平均迭代次数**
- 完成任务所需的Thought-Action循环数
- 越少越好(通常3-5次)

**性能指标**:

**4. 端到端延迟**
- 从用户输入到最终输出的时间
- 目标: <10秒

**5. Token消耗**
- 每次任务的平均token数
- 成本控制关键

**质量指标**:

**6. 答案准确性**
- 人工评估或Golden Dataset对比
- 目标: >80%

**7. 用户满意度**
- 点赞/点踩比例
- NPS(净推荐值)

**鲁棒性指标**:

**8. 错误恢复率**
- 失败后重试成功的比例
- 目标: >70%

### 评分标准
- 列出6个以上指标得6分
- 每个指标解释清晰得1分(共6分)
- 给出目标值得2分
- **总分**: 14分,≥11分为合格

## 题目10: LangChain vs Spring AI for Agent ⭐⭐

**题型**: 简答题  
**难度**: ⭐⭐⭐  
**知识点**: 框架选型

### 问题
LangChain(LangGraph)和Spring AI在Agent开发上各有什么优劣?

### 参考答案
**LangChain/LangGraph**:
- **优势**:
  - Python生态,社区活跃
  - LangGraph原生支持多Agent工作流
  - 丰富的预建Tools和Agents
  - 文档完善,示例丰富
- **劣势**:
  - Java项目需跨语言调用
  - 生产环境稳定性待验证

**Spring AI**:
- **优势**:
  - Java原生,与Spring生态无缝集成
  - 企业级特性(事务、安全、监控)
  - Function Calling类型安全
  - 适合已有Java团队
- **劣势**:
  - 生态较新,社区较小
  - Agent高级功能仍在开发

**选型建议**:
- **Python团队/快速原型**: LangChain
- **Java企业项目**: Spring AI
- **混合架构**: LangChain做Agent,Spring Boot做业务,通过HTTP/gRPC通信

### 评分标准
- LangChain优劣势各2分(共4分)
- Spring AI优劣势各2分(共4分)
- 选型建议合理得2分
- **总分**: 10分,≥8分为合格
